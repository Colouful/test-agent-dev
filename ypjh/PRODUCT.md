# PRODUCT.md — 错题本：产品上下文

> 给 AI Agent 的产品判断手册。读完这个文件，Agent 应该能回答："我现在要加的这个功能，符合这个产品的方向吗？"

---

## 为什么存在

学生做错题本的真正障碍不是"懒得整理"，是"整理成本太高"——拍照、手抄、分类、复习提醒，每一步都要人工操作。错题本用 AI 识别消灭拍照→录入这一步，让学生只需确认一次，后续复习自动推送。

这是一个 **课程练手台架**，全班做同一个应用，目的是走完从需求设计到云端部署的完整 AI 工程工作流，带走一套工程判断力。功能不是越多越好，工作流完整比功能丰富更重要。

---

## 目标用户

中学生、大学生、备考人员。**初期个人使用**，无多用户协作需求。

---

## 成功标准（MVP）

走通这条完整闭环即为成功：

> 上传图片 → Bedrock 识别 → 用户确认 → 结构化入库 → 列表查看

不需要功能完备，需要每一步都**正确且可靠**。

---

## 功能优先级

| 优先级 | 功能 | 说明 |
|--------|------|------|
| **P0** | 拍照识别错题 | AI 识别主线，MVP 核心 |
| **P0** | 错题存储 + CRUD | 按学科/难度分类，手动录入 + 编辑 |
| **P1** | SM-2 复习推荐 | 字段已预留（ease_factor / next_review_at），算法后实现 |
| **P2** | 打印排版 | 生成可打印错题卡片，课程最后阶段 |

---

## 非目标（Agent 不应自行添加这些功能）

| 非目标 | 原因 |
|--------|------|
| 缓存层（Redis 等） | 个人使用量级，无需缓存；监管要求识别结果实时处理 |
| 多用户协作 / 分享功能 | 错题是个人隐私数据，初期不做共享 |
| 实时 WebSocket 推送 | MVP 同步 HTTP 已满足需求，队列/WS 是升级路径 |
| 批量导入（拍照以外的方式） | MVP 只做拍照识别一条主线 |
| 社交功能（点赞、评论） | 与产品定位无关 |
| 管理后台 | 个人工具，不需要运营视角 |
| 真实 Bedrock 接入（MVP 阶段）| MVP 用 `MOCK_BEDROCK=true`，真实接入是下一阶段 |
| SM-2 算法实现（MVP 阶段）| 字段预留，算法不在 MVP 范围 |
| PDF/打印（MVP 阶段）| P2 功能，MVP 不实现 |

**Agent 判断标准**：如果一个功能不在 P0 列表里，先问用户确认再动手，不要自行判断"顺手加一下"。

---

## 关键产品约束

- 识别结果**必须经用户确认**才能入库，不得自动入库（低信任 AI 输出）
- 每道错题与 `user_id` 绑定，**数据严格个人隔离**，不得出现跨用户可见
- 图片是题目的一部分（几何图/电路图），**原图永久保留**，不得在删除题目时一并删除
- 识别是**非确定性 AI 输出**，系统设计必须为"识别结果可能错误"兜底，而非假设识别总是正确
