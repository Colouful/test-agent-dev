# 规则分级标准（Severity Guide）

## 三级定义

| 级别 | Agent 默认行为 | 偏离处理 |
|------|--------------|---------|
| **MUST** | 立即停下来，询问用户是否确认 | 不得在未获确认的情况下继续执行 |
| **SHOULD** | 默认遵守，自动执行 | 偏离时在代码注释或 commit message 里说明理由，然后继续 |
| **CAN** | 倾向遵守，直接执行 | 偏离无需说明，不留记录 |

---

### MUST — 硬约束（红线）

违反即拒绝。Agent 想要违反时，必须**立即停下来询问用户**，不得闷头执行。
用户明确确认后方可继续，且必须在 commit message 中注明"经用户确认偏离 Rn"。

适用场景：
- 安全边界（数据隔离、认证绕过）
- 数据完整性（不可逆的错误存储方式）
- AI 输出的安全门（识别结果未校验直接入库）

MUST 规则要**极少、极精**。如果每条都是 MUST，等于没有 MUST。

### SHOULD — 强建议（默认遵守）

默认遵守，有充分理由可以偏离。偏离时：
1. 在偏离处写代码注释说明原因（一行即可）
2. 在 commit message 里注明"偏离 Rn，原因：…"
3. 继续执行，不需要等待用户确认

适用场景：
- 架构分层规则（路由不直接操作 DB）
- 性能最佳实践（async IO）
- 可维护性规范（函数长度、命名）

### CAN — 可选偏好（灵活）

倾向遵守，但根据具体情况可灵活调整。偏离无需说明、不需要注释、不需要等待确认。

适用场景：
- 代码风格细节
- 可选的性能优化
- 开发阶段的权宜之计

---

## 判断方法

遇到不确定的规则，问自己三个问题：

1. **如果违反，会不会让用户数据泄漏、丢失或被污染？** → MUST
2. **如果违反，代码还能跑，但维护起来会越来越痛？** → SHOULD
3. **如果违反，只是看起来不够优雅？** → CAN

---

## 原则

> "全是 MUST 等于没有 MUST。"
> MUST 是红线，不是品味清单。
