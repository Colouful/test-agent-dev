# Lab 12 改进版 — 给你的 Claude Code 建一道不会被骗的验证防线

> **设计原则**：每层产出都是学员课后能直接用的真实系统。参照 SwarmAI 实际经验（Voice Mode 10/10 翻车事故 / pipeline confidence illusion / 质量收敛数据），让学员亲手体验"被骗"的代价和"不被骗"的方法。
>
> **时长**：50 分钟
> **工具**：Claude Code（唯一工具）
> **前置**：harness repo 已有 `spec/`（M8）、`src/`（M9 识别功能初版）、`.claude/agents/`（M10）
> **产出承诺**：课后你有一个 `ci/verify.sh`——它能当 M13 loop 的无人值守停止条件，也能在日常开发中一键验证代码质量
>
> **层次设计**：
> - **Layer 1 — 亲手被骗**（15 min）：复现同义反复陷阱，体验"测试全绿但代码是错的"——震撼建立动机
> - **Layer 2 — 建防线**（20 min）：写三类断言（结构/契约/统计），学会验证非确定 AI 输出——核心技能
> - **Layer 3 — 封装终点线**（15 min）：整合成机器友好的 verify.sh + 特征快照——直接成为 M13 loop 的裁判

---

## 设计哲学：从 SwarmAI 学到什么

SwarmAI 的验证体系（P0 缺陷从 1.0→0.0/版）给了三个关键教训：

1. **"自信"和"正确"是负相关的** — SwarmAI Voice Mode 10/10 翻车事故（`post-mortems/01-pipeline-confidence-illusion.md`）：pipeline run `6823b0d4` 所有检查全绿、置信度 10/10，但实际音频是空的——Polly Zhiyu 引擎静默产出了 0 字节音频，happy-path 测试根本没验证音频内容。教训：验证必须验**交付物本身的属性**，不能只验"流程是否走完了"。

2. **同义反复是 AI 协作的头号杀手** — 当同一个 agent 既写实现又写测试，它的"误解"会同时体现在两处——代码错了，测试也错了，结果全绿。SwarmAI 用 `test-author` 与 `implementer` 的上下文隔离（两个 Claude CLI 子进程，不共享对话）来打破这个闭环。学员版：用 M10 的 subagent 机制做同样的隔离。

3. **质量收敛的秘密是 gates，不是 carefulness** — SwarmAI 的 `CONVERGENCE.md` 数据：bug 抓得更多、更早、更轻。靠的不是"agent 更小心了"，而是门控越来越多（结构断言挡住结构错误，契约断言挡住逻辑错误，统计断言挡住漂移）。Carefulness doesn't scale, gates do。学员版：三层断言就是三道门。

## Layer 1 — 亲手被骗：同义反复 A/B 实验 (15 min)

### 目标
亲手复现 AI 协作中最隐蔽的陷阱：同一个 agent 既误解需求、又写出"证明自己正确"的测试。然后用隔离打破它。

### 课程概念
- 同义反复 Tautological Test（Slide 2）
- 武器一：测试独立性隔离（Slide 5）
- "它全绿了但它是错的"——AI 协作特有的验证盲区
- **SwarmAI 对标**：test-author vs implementer 子进程隔离 + Voice Mode 事故的"所有检查全绿但产出是空的"

### Step 1.1 — 实验 A：复现陷阱（5 min）

```bash
claude
> 我需要你一口气完成两件事：
> 1. 实现识别功能（从作业照片中识别出题目）
> 2. 为你的实现写单元测试
>
> 需求大意：从上传的作业照片里识别出题目。
> 注意：不要先读 spec/ 目录。根据你对"识别题目"的理解直接写。
```

**预期**：agent 很可能把需求理解为"识别图里所有题"（常识推断），漏掉 spec 里"只提取标错的题"这条边界。然后写出的测试会验证"输入 5 题图片 → 返回 5 题"——这恰好证明了它的（错误的）理解是对的。测试全绿。

```bash
# 验证"全绿但错了"
> 运行你刚写的测试

# 预期：全部通过 ✅ — 但这是个陷阱！

> 现在读一下 spec/recognize/requirements.md，找到关于"识别哪些题"的具体要求

# 预期：学员发现 spec 说"只提取标错的题"，而实现是"提取所有题"
```

### Step 1.2 — 实验 B：用隔离戳破（10 min）

```bash
# 定义 test-author 子代理（如果 M10 已有可跳过）
> 帮我创建 .claude/agents/test-author.md：
>
> ---
> name: test-author
> description: 只依据 spec 文档写测试。绝不读实现代码。
> tools: Read, Write, Grep
> ---
>
> # 角色：独立测试作者
>
> ## 职责
> 1. 只读 spec/ 目录下的需求文档
> 2. 根据需求边界写测试用例——特别关注边界条件和"不该做什么"
> 3. **绝不读 src/ 目录** — 你对实现一无所知，这是故意的
>
> ## 关注点
> - 需求里的"不"字（不应该、只有、除非、仅当）
> - 边界条件（空输入、极端值、混合情况）
> - 输出格式约束

# 用隔离子代理重写测试
> 用 test-author，只依据 spec/recognize/requirements.md 写测试

# 跑测试 — 看 Step 1.1 的实现能不能通过
> 运行 test-author 写的测试
```

**预期**：test-author 读了 spec，会写出"喂对错混排图 → 断言只返回标错的题"这种测试。Step 1.1 的"返回所有题"实现**必然失败**。

**关键教学时刻**：

```bash
# 讲师问（or agent 引导讨论）
> 为什么实验 A 全绿，实验 B 失败了？两次用的是同一份实现代码。
```

> 答：A 里测试和实现来自同一个"理解"——误解被两边共享了，所以它们"互相证明"。B 里测试来自 spec（地面真相），实现来自 agent 的理解——两者不匹配时，测试就能发现问题。这就是同义反复：**让犯错者自己验证自己，永远验不出错**。

**SwarmAI 对标讲解**（讲师用）：
> SwarmAI Voice Mode 事故完美诠释了这一点：pipeline 的"验证"只检查了"流程有没有正常走完"（exit code 0、文件有没有生成），没验证交付物本身的属性（音频是否有内容）。结果 happy-path 全绿，但产出是空音频。
>
> 教训：验证必须独立于实现者的假设。用隔离来保证这种独立性。SwarmAI 现在的做法：test 和 impl 跑在不同的 Claude CLI 子进程中，上下文完全隔离——它们能"互相发现对方的盲区"恰恰因为它们不共享认知。

### Done 标准
- [ ] 实验 A 复现了同义反复（全绿但错误，或手动注入后复现）
- [ ] 实验 B 的隔离测试成功揭露了错误实现
- [ ] 学员能口头解释"为什么同一个 agent 写实现+测试是危险的"
- [ ] test-author 子代理可复用（课后所有测试都可以这样写）

## Layer 2 — 建防线：三类断言验证非确定输出 (20 min)

### 目标
为识别功能（非确定 AI 输出）写三层断言：结构 → 契约 → 统计。学会"验结构与契约、不验内容"的核心方法论。

### 课程概念
- 武器二：验证非确定输出（Slide 6）
- 三类断言各自守什么线（结构=格式合规、契约=边界规则、统计=驯服波动）
- 关键判断："哪条线必须确定、哪条容忍波动"——这是人划的，AI 划不了
- **SwarmAI 对标**：`verify.sh` 中的分层（L1 静态 / L2 结构+契约 / L3 统计+快照）+ 验证金字塔

### Step 2.1 — 结构断言：输出格式必须合规 (5 min)

```bash
claude
> 帮我为识别功能写结构断言（写入 tests/recognize.structure.test.ts 或等价文件）：
>
> 不管识别出什么内容，以下结构属性必须 100% 满足：
> 1. 返回值是合法 JSON 数组
> 2. 数组每个元素必须有 content（string）、subject（string）、confidence（number）字段
> 3. confidence 范围是 [0, 1]
> 4. content 非空字符串
> 5. 当输入为空图片时，返回空数组（不报错）
>
> 使用 mock 输入（构造固定的识别返回值来验证结构约束）。
> 这些断言跟"识别了什么"无关，只验格式。它们不调真实 AI，所以秒级完成。

# 跑一下确认
> 运行结构断言测试
```

**关键认知**：结构断言是**确定的**——不管 AI 模型怎么变，格式必须合规。这是最便宜（mock）、最快（毫秒级）、最不可能误报的验证层。

### Step 2.2 — 契约断言：业务边界不可越 (8 min)

```bash
> 帮我写契约断言（tests/recognize.contract.test.ts），基于 spec/recognize/requirements.md 的边界规则：
>
> 契约 1 — "只提取标错的题"：
>   构造一个 mock 输入（5 题，其中 3 对 2 错），断言只返回 2 题
>
> 契约 2 — "低置信度标待确认"：
>   构造一个 confidence=0.5 的识别结果，断言该题 status="pending_review" 而非直接入库
>
> 契约 3 — "0 错题时返回空+提示"：
>   构造一个全对（0 错题）的输入，断言返回空数组 + 提示信息"未发现错题，建议重拍"
>
> 所有输入都是 mock 构造的。这些断言验的是"业务规则是否被遵守"，不是"AI 识别了什么"。
> 它们和结构断言一样确定（mock 输入 → 确定输出）。

# 跑一下
> 运行契约断言测试

# 可能的结果：如果 Step 1 的实现是"返回所有题"，契约 1 会失败 — 这正好！
```

**关键教学时刻**：
> 讲师问："这些契约断言和 Step 1 test-author 写的测试有什么关系？"
> 答：test-author 从 spec 推导出的测试，本质就是契约断言——验的是"需求边界有没有被遵守"。契约断言把这些用例**固化成可重复执行的代码**，之后每次改动都自动跑。

### Step 2.3 — 统计断言：驯服不确定性 (7 min)

```bash
> 帮我写一个统计断言（tests/recognize.stats.test.ts）——这个不同，它接受波动：
>
> 对同一张样例图（tests/fixtures/sample-math.jpg 或一个 mock 输入）：
> 1. 运行识别 N=5 次（如果没真实 AI 可用就用带随机扰动的 mock）
> 2. 断言：
>    - 识别出的题目数众数 ≥1（不是每次一样，但大多数时候能识别出题）
>    - 平均 confidence ≥ 0.6（整体质量达标，允许单次波动）
>    - 每次返回都通过结构断言（格式永远不能错）
> 3. 不断言逐字内容（"题干是什么"——这会 flaky）
>
> 注明：这个测试真调 AI（或模拟真调），比较慢和贵。
> 在 ci/verify.sh 里它应该被标记为"低频单独跑"，不放进 loop 的每轮主终点线。
```

**关键教学时刻**：
> 讲师问："为什么统计断言不放进 loop 的每轮终点线？"
> 答：两个原因。第一，**经济学**——loop 每轮都跑一次验证，如果统计断言真调 AI，就是双倍烧钱（改代码烧一次 + 验证烧一次）。第二，**确定性**——统计断言天生有波动（5 次里可能有 1 次偏低），让它当 loop 停止条件会导致 loop 误判"还没修好"。主终点线只放确定的结构+契约断言。

**SwarmAI 对标讲解**（讲师用）：
> SwarmAI 的验证金字塔：
> - 底层（每次都跑）：lint + type check + 结构断言 — 秒级、免费、确定
> - 中层（每次都跑）：契约断言（mock）— 秒级、免费、确定
> - 上层（低频）：统计断言 + golden set — 慢、贵、有波动
>
> Voice Mode 事故的修复就是在中层加了一条："音频文件 size > 0 且 duration > 0"——这是一条结构断言，而不是"音频质量好不好"的统计断言。最致命的 bug 往往被最便宜的断言挡住。
>
> M12 把统计断言移出主终点线的决定，直接为 M13 省钱——讲师要把这条因果链讲透（"两头烧钱"）。

### Done 标准
- [ ] 三类断言各至少一条，分文件存放
- [ ] 结构+契约断言是确定的（mock，不 flaky）
- [ ] 统计断言验分布/区间不验逐字内容
- [ ] 学员能划清"必须确定 vs 容忍波动"的线并说出理由
- [ ] 理解"为什么统计断言不放进 loop 主终点线"

## Layer 3 — 封装终点线：verify.sh + 特征快照 (15 min)

### 目标
把 Layer 2 的断言整合为一个机器友好的终点线脚本 + 一份特征快照基线。这个 `ci/verify.sh` 明天直接成为 M13 loop 的停止条件。

### 课程概念
- 武器四：机器友好的终点线（Slide 8）
- 特征测试防 prompt 漂移（Slide 7）
- 验证金字塔：分层 + 标注（Slide 10）
- **SwarmAI 对标**：`CONVERGENCE.md` 的 "gates > carefulness" + 分层 verify pipeline

### Step 3.1 — 整合 verify.sh (7 min)

```bash
claude
> 帮我创建 ci/verify.sh — loop 的终点线脚本，设计原则：
>
> 1. 退出码语义：0=通过，1=失败（可修），2=环境错误（不可修，叫人）
> 2. 失败时 stdout 输出富上下文信息（告诉 loop 的 agent 具体什么错了，怎么修）
> 3. 分层执行 + 快速失败（前一层挂了不跑后面）：
>
> ```bash
> #!/usr/bin/env bash
> set -uo pipefail
>
> echo "=== L1: 静态检查 ==="
> # TypeScript 类型检查 + ESLint
> # 失败时：echo "VERIFY_FAIL stage=L1 ..." && exit 1
>
> echo "=== L2: 结构 + 契约断言（确定，mock，loop 主终点线）==="
> # 运行 tests/recognize.structure + tests/recognize.contract
> # 这是 loop 每轮都跑的核心
> # 失败时：echo "VERIFY_FAIL stage=L2-contract ..." && exit 1
>
> echo "VERIFY_OK — 主终点线通过"
> exit 0
>
> # === 以下不在主终点线内（loop 不跑，单独低频触发）===
> # L3: 统计断言（真调 AI，慢且贵）
> # L4: 特征快照对比（检测 prompt 漂移）
> ```
>
> 4. 确保脚本可执行
> 5. 注释清晰标注"loop 跑这些" vs "人/CI 低频跑这些"
```

**验证**：

```bash
# 正常通过
./ci/verify.sh; echo "exit=$?"    # 预期：VERIFY_OK, exit=0

# 故意改坏实现（删掉"只返回错题"的逻辑）
# 再跑
./ci/verify.sh; echo "exit=$?"    # 预期：VERIFY_FAIL stage=L2-contract, exit=1
                                   # stdout 包含哪条契约失败、在哪个文件
```

### Step 3.2 — 特征快照防漂移 (8 min)

```bash
> 帮我创建特征快照机制：
>
> 1. 准备 tests/fixtures/ 目录，放入 2-3 张代表性输入（或 mock）：
>    - 数学题（期望识别出公式）
>    - 语文题（期望识别出文字）
>    - 混排题（3 对 2 错，期望只返回错的）
>
> 2. 创建 tests/snapshot.sh 脚本：
>    - 对每张 fixture 运行识别
>    - 记录结构特征到 tests/baseline.json：
>      - 每张图：识别题数、confidence 均值、字段完整性
>      - 注意：只记结构特征，不记逐字内容
>    - 如果 baseline.json 已存在：对比新结果 vs 基线
>      - 结构特征变化 >20% 的项标为 ⚠️ DRIFT
>      - 输出哪些图的特征漂移了
>
> 3. 这个脚本不在 verify.sh 主终点线里（因为它可能真调 AI）
>    但在"改了识别 prompt 后"应该手动跑一次，检查有没有"改 A 坏 B"
```

**演示漂移检测**：

```bash
# 拍基线
./tests/snapshot.sh --baseline

# 修改识别 prompt（模拟"为数学优化"的改动）
> 在识别 prompt 中加一句"特别注意 LaTeX 公式的正确性"

# 重新跑快照对比
./tests/snapshot.sh --compare

# 预期：数学那张可能改善了，但语文那张可能被带歪——漂移被抓到
```

**关键教学时刻**：
> 讲师问："为什么快照记'结构特征'而不记'识别出的具体文字'？"
> 答：具体文字每次都可能不同（AI 非确定性），记下来就永远在报 diff——噪声太多看不到信号。结构特征（题数、置信度档位、字段完整性）在正确的实现下是稳定的。漂移了说明行为真的变了，不是正常波动。

**SwarmAI 对标讲解**（讲师用）：
> SwarmAI 的 `golden_set` eval 就是工业版特征快照：
> - 固定一批代表性输入
> - 跑完后不对比"逐字输出"，而对比"输出满足哪些属性"
> - 属性通过率下降 → 触发告警
>
> 学员版简化为 bash 脚本 + JSON 基线——思想一样，但无需 eval 基础设施。当项目长大、prompt 改动频繁时，可升级为 CI 里的 eval job（M14 预告）。

### Done 标准
- [ ] `ci/verify.sh` 退出码清晰（0/1/2），失败信息富含上下文
- [ ] 主终点线只含 L1+L2（确定、mock、秒级）
- [ ] 统计+快照显式标注为"低频单独跑"
- [ ] 特征快照记录结构特征，非逐字内容
- [ ] 改 prompt 后快照能检测出漂移
- [ ] verify.sh 明天可直接被 M13 loop 调用

**课后使用指南**（一句话）：日常改代码后跑 `./ci/verify.sh` 一键检查（秒级）；改了 AI prompt 后跑 `./tests/snapshot.sh --compare` 检查漂移；写新功能时用 `test-author` 子代理独立写测试——你的验证体系会比"手动跑一下看看"可靠一万倍。

## Lab 复盘讨论 (3 min)

1. **同义反复的本质**：实验 A 里，为什么"识别所有题"那个 bug 能骗过测试？和邻座讨论——你们的 test-author 分别抓住了什么边界条件？

2. **划线的判断（核心）**：你给识别输出划的"必须确定 vs 容忍波动"那条线在哪？如果把"识别出的题数"也设成容忍波动——loop 可能修出什么？（答：loop 可能删掉某些题的识别逻辑来"通过"统计断言的下界——因为"少识别几道"也在区间内）

3. **经济学**：你的 verify.sh 主终点线全是 mock 断言。如果 loop 每轮真调 Bedrock——错题本 loop 是两头烧钱。M12 这个"移出主终点线"的设计，省了多少？（答：假设 loop 跑 5 轮收敛，每轮真调一次 Bedrock = 5 次额外 API 调用。mock 契约断言 = 0 次。5x 的成本差）

4. **通向 M13**：你写的 verify.sh 敢不敢让一个无人值守的 loop 当唯一裁判？还缺什么？（答：缺"人最终审一眼 PR"的 gate——verify.sh 当裁判，人当最后关口。两者缺一不可）

## 与原版实验的关键改变

| 维度 | 原版 | 改进版 |
|------|------|--------|
| **体验震撼** | 直接教方法（写测试→跑通） | Layer 1 先"被骗"再"解骗"——动机建立在体验上 |
| **层次感** | 5 步并列，无优先级 | 3 层递进：被骗→防骗→封装，每层 self-contained |
| **学员记忆** | "AI 帮我写了测试"（浅） | "我亲手被同义反复骗了一次"（震撼，记得住） |
| **产出复用** | 需大量改造 | verify.sh 明天 M13 直接调用，课后日常用 |
| **SwarmAI 对标** | 概念引用 | 每步标注 SwarmAI 事故/设计 + 讲师可展开 |
| **与 loop 衔接** | Step5 提一句"明天用" | 全程贯穿"这是 loop 的裁判"的设计视角 |
| **非确定性处理** | 三类断言教学为主 | 加入"为什么不放进 loop 主终点线"的经济学判断 |

## 讲师备注

### Layer 1 的 A/B 实验是灵魂

如果课堂时间紧，可以压缩 Layer 3（verify.sh 部分学员跟着做即可），但 **Layer 1 的 A/B 对比实验绝不能省**。原因：

1. "被骗"的体验是建立动机的唯一可靠方式。如果你跳过这步直接教"用隔离写测试"——学员会想"反正 AI 帮我写测试也能用"，动机为零
2. 这是 AI 协作特有的问题（传统开发里程序员不会"误解需求后写出自洽的证明"），必须亲手体验才能真正理解
3. SwarmAI Voice Mode 事故是工业级的验证——你可以在实验 A 之后展示这个事故，让学员看到"全绿但空产出"在生产环境也会发生

### 实验 A 的容错

如果 agent 这次恰好理解对了（概率约 30%——因为"只提取错题"是比较 niche 的需求），有两个备案：

1. 手动注入：让学员把实现改成"返回所有题"，然后跑 B 的测试——同样能展示隔离的价值
2. 换需求：用另一个容易被误解的需求（如"识别结果只包含学科内的题目"——agent 可能漏掉"跨学科题应标记"这条）

### 关于 Bedrock 依赖

整个 Lab 设计为**不依赖 Bedrock 真调**——结构/契约断言全用 mock。统计断言和快照如果没有真 AI 可调，用"带随机扰动的 mock"模拟即可。这保证了：
- Lab 在任何环境都能跑通
- 学员理解"mock 确定性"和"真调波动性"的区别
- 不烧钱

讲师准备一组 mock 识别返回值（含对错混排/低置信度/0 题三种）作为安全网。
